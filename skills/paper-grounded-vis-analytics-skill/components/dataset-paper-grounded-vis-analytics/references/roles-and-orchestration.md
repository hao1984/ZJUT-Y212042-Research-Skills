# Roles And Orchestration

This skill runs as a staged Mechanism-E visual analytics pipeline. Each stage is an AI role with a specific prompt contract, input artifact, and output artifact. When only one model is available, the model must still switch roles explicitly and write stage artifacts before moving on.

## Role 0: Orchestrator

Purpose: keep the pipeline staged, auditable, and resistant to dashboard collapse.

Actions:

- Create `runs/<run_name>/`.
- Record dataset path, user goal, auxiliary files, and paper corpus paths.
- Load only the English prompt needed for the next role from `references/source-prompts-english/`.
- Use `references/source-prompts-verbatim/` only for source-fidelity audit.
- Keep deterministic helpers separate from model design decisions.

Output:

- `artifacts/orchestration_manifest.json`
- stage-specific file paths and role trace

## Role 1: Data Loading And Standardization Agent

Prompt:

- `source-prompts-english/stage1a_load_v0.1.md`

Inputs:

- user dataset path
- optional auxiliary files
- user description

Responsibilities:

- Load CSV, TSV, JSON, JSONL, parquet, Excel, directories, or related tabular sources when possible.
- Choose a primary table and preserve lookup/metadata tables as auxiliary tables.
- Never replace data with mock, sample, or synthetic rows.
- Fail fast with explicit errors if the primary table cannot be read.

Outputs:

- `stage1_load/loading_manifest.json`
- `stage1_load/standardized_object.yaml`
- `stage1_load/data/primary.parquet` or another explicit standardized object
- `stage1_load/data/auxiliary/*` when auxiliary sources exist

## Role 2: Data Detective / Profile Agent

Prompts:

- `source-prompts-english/data_profile_v0.1.md`
- `source-prompts-english/DATA_PATTERN_MINING_PRINCIPLES.md`

Inputs:

- standardized object
- user description
- data hints

Responsibilities:

- Discover what the data says, not just what columns exist.
- Separate user assumptions from data evidence.
- Produce field profiles, quality facts, pattern evidence, candidate research questions, and anti-dashboard checks.
- Treat generic schema facts as profile context, not mined patterns.

Outputs:

- `stage1_profile/data_profile.yaml`
- `stage1_profile/pattern_evidence.json`
- optional `intermediate_plots/*` only when they help verify a candidate pattern

## Role 3: Narrow Pattern Mining Agent

Prompts:

- Primary: `source-prompts-english/stage1b_mining_narrow_v0.2.md`
- Compatibility: `source-prompts-english/stage1b_mining_narrow_v0.1.md`
- JSON contracts: `source-prompts-english/stage1b_mining_brief_json_v0.4.md`, `source-prompts-english/stage1b_mining_json_response_v0.3.md`

Inputs:

- basic profile
- data profile
- user description

Responsibilities:

- Mine data-specific patterns, subgroup contrasts, conditional/multidimensional couplings, boundary cases, and relation structures.
- Build a pattern graph.
- Propose research questions that can become visual analysis systems.
- Reject RQs that only produce overview dashboards, field browsers, KPI cards, or top-N summaries.

Outputs:

- `stage1_mining/mining_result.json`
- `stage1_mining/pattern_graph.json`
- optional `stage1_mining/targeted_checks.json`

## Role 4: Paper Retrieval And Grounding Agent

Prompt support:

- `paper-guidance-policy.md`
- local paper corpus generated by `$paper-mineru-scholar-index`

Inputs:

- data profile
- mined patterns
- candidate research questions
- `data/index.db`
- `data/papers/**/meta.json`
- `data/papers/**/paper.md`

Responsibilities:

- Search the local paper index using dataset semantics, field names, pattern descriptions, and RQs.
- Read retrieved `meta.json` and `paper.md` when available.
- Extract reusable visual structures, interaction models, evidence panels, provenance idioms, and layout principles.
- State why each borrowed idea applies to the current dataset.
- Mark unused references with reasons.

Outputs:

- `stage2_references/paper_reference_digest.json`
- `stage2_references/paper_reference_report.md`

## Role 5: Mechanism-E Idea Architect

Prompt:

- `source-prompts-english/data_profile2idea_v0.1.md`

Inputs:

- profile
- mined patterns
- paper reference report
- user goal

Responsibilities:

- Select a strong analysis target.
- Convert patterns into a designable VIS research idea.
- Preserve the five Mechanism-E contracts:
  - `analysis_target`
  - `data_task_encoding_mapping`
  - `why_not_dashboard`
  - `coordinated_workspace`
  - `exploration_affordance`
- Keep the idea grounded in real data and retrieved paper design precedents.

Outputs:

- `stage2_idea/idea.yaml`
- `stage2_idea/e_idea_contract.yaml` or `.json`
- `stage2_idea/rq_selection.md`

## Role 6: Visual System Designer

Prompts:

- `source-prompts-english/stage3bc_visual_frontend_builder_v0.1.md`
- `source-prompts-english/demo_builder_e_v0.1.md`
- `frontend-contract.md`
- `llm-frontend-builder-guidelines.md`

Inputs:

- idea contract
- paper reference report
- real dataset payload
- mined pattern evidence

Responsibilities:

- Write `artifacts/design_spec.md` before frontend code.
- Define the dominant visual object, visual grammar, view graph, shared state, linked interactions, data payload strategy, and first-screenshot intent.
- Avoid fixed templates and generic dashboard layouts.
- Use paper guidance as design reasoning, not as copied UI.

Outputs:

- `stage3_visual_spec/visual_system_spec.json`
- `artifacts/design_spec.md`

## Role 7: Frontend Builder

Prompts:

- `source-prompts-english/demo_builder_e_v0.1.md`
- `source-prompts-english/stage3bc_visual_frontend_builder_v0.1.md`

Inputs:

- design spec
- visual system spec
- dataset payload

Responsibilities:

- Build a static single-page visual analytics system by default.
- Draw real rows or real aggregations.
- Implement cross-view interactions.
- Keep first screen usable at `1920x1080` and `1440x810`.
- Provide visible provenance.

Outputs:

- `app/index.html`
- `app/style.css`
- `app/main.js`
- `app/data/*`
- `artifacts/frontend_build_report.md` or `BUILD_REPORT.md`
- `demo_metadata.json` when requested by the prompt

## Role 8: Internal E Critic

Prompt:

- `source-prompts-english/e_critic_v0.1.md`

Inputs:

- stage artifacts
- screenshots or browser smoke output
- HTML/CSS/JS
- profile, idea contract, and paper reference report

Responsibilities:

- Score with evidence.
- Identify hard-fail flags.
- Independently check real-data provenance, data-pattern fidelity, idea fidelity, paper-guidance implementation, first-screen visual quality, viewport fit, and coordinated workspace behavior.
- Do not trust the app's own self-check without evidence.

Outputs:

- `artifacts/e_critic_stage1.json`
- `artifacts/e_critic_stage2.json`
- `artifacts/e_critic_stage3.json`
- `artifacts/visual_quality_review.json`

## Role 9: Repair Agent

Prompt:

- `source-prompts-english/stage3e_repair_v0.1.md`

Inputs:

- critic findings
- browser smoke report
- frontend files
- idea and design specs

Responsibilities:

- Repair only failed gates.
- Preserve the core analysis object and idea contract.
- If real data forces a design deviation, record it in build metadata.
- Re-run smoke checks after repairs when possible.

Outputs:

- repaired `app/*`
- updated build report
- updated smoke/quality reports

## Deterministic Helper Policy

Use scripts as support, not as the creative frontend generator:

- `scripts/prepare_dataset_vis_brief.py`: prepare dataset profile, mined-pattern seed, paper reference digest, payload files, and an LLM design brief.
- `scripts/browser_smoke_check.cjs`: verify basic frontend render, viewport fit, nonblank marks, and coordinated workspace signals.
- `scripts/run_dataset_vis_pipeline.py`: legacy fallback only. Do not use it as the default final app generator.

## Run Completion Criteria

A run is complete only when:

- all major stages have written artifacts,
- the frontend is runnable,
- primary marks come from real data,
- paper guidance is visible in the design reasoning,
- at least two analytical views are linked,
- critic hard-fail flags are cleared or explicitly documented,
- the user receives a path or URL that can be opened for demonstration.
